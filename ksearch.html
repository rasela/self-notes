Local-first tooling

ksearch
search by meaning

Semantic search over my notes — no cloud, no key, no cost

Notes → Embed → Index → Recall

One line — ksearch finds notes by what they mean, not the words they contain. Ask "how do dealers hedge short gamma" and it returns the right section even if that exact phrase appears nowhere. Runs entirely on my own machine off a small local model.

What It Is

Meaning, not keywords

grep matches letters. Search a paraphrase — "quietly absorbing supply into strength" — and if my note said "hidden accumulation on the bid," grep returns nothing. The idea is identical; the words don't overlap, so keyword search is blind to it.

Semantic search matches meaning. Every chunk of text becomes a vector — a list of ~768 numbers that encodes what it's about. Two passages that mean the same thing land near each other in that space, whatever words they used. The query becomes a vector too; the nearest chunks are the answer.

So ksearch is a thin tool over that idea, pointed at my own notes: a small program that turns my notes into vectors once, then answers questions by finding the closest ones.

grep = exact words semantic = concepts paraphrase-proof ranked by closeness

Why Local

The whole thing runs on my laptop

The vectors ("embeddings") come from a model called nomic-embed-text running under Ollama — a runner that serves local models over plain HTTP on localhost:11434. The model is ~274 MB on disk and holds ~0.3–1 GB of RAM while working.

Nothing leaves the machine. That buys four things at once:

01
PrivateMy trading notes and journals never touch a third-party API. The corpus stays on disk.
02
FreeNo per-token bill. Re-index the whole set as often as I like; it costs electricity, not money.
03
OfflineWorks on a plane. No network dependency, no key to rotate, no rate limit.
04
SmallAn embedding model is tiny next to a chat model — it only has to understand text, not generate it. Fits comfortably in spare RAM.

Key distinction — an embedding model reads and returns a vector. A chat model (Llama, etc.) writes and needs many times the memory. ksearch only needs the cheap, read-only kind.

How It's Built

One Python file, standard library only

The entire tool is a single script — semsearch.py — using nothing but Python's standard library. No torch, no virtualenv, no packages to install. The only outside call is HTTP to the local Ollama server.

It has exactly two jobs: build the index once, and query it many times.

The pipeline

# BUILD  (run once, and after notes change)
knowledge/*.html
   │  strip <script/style/nav/header/footer>
   │  split at <h1–h3> headings  →  chunk ≤1500 chars
   │  embed each chunk  (POST localhost:11434)
   ▼
index.json  =  [{ file, heading, text, vec[768] }, ...]

# QUERY  (every search)
"how do dealers hedge short gamma"
   │  embed the query  (same model)
   │  cosine-similarity vs every stored vec
   │  sort, take top-k
   ▼
[1] 0.74  absorption.html › Dealer hedging
[2] 0.71  footprint.html  › Gamma & flow
    ...

The clever part is boring on purpose. Chunking by heading keeps each vector about one idea, so a hit points at a section, not a whole page. Cosine similarity is just the angle between two vectors — smaller angle, closer meaning — and it's a few lines of plain math, no library needed.

1
Clean

HTML in, readable text out. Drop tags and boilerplate so only the note's words get embedded.

2
Chunk

Cut each page at its headings, then split anything over ~1500 chars on sentence boundaries. One chunk ≈ one idea.

3
Embed

Send every chunk to Ollama, get back a 768-number vector. Store text + vector together in one JSON file.

4
Rank

At query time, embed the question, score it against every stored vector by cosine, print the closest few with page › heading.

Why it lives off the notes repo — the notes site auto-publishes its whole folder to the web. The tool + index sit in a separate local directory (~/.local/share/knowledge-semsearch/) so the script and my vectors never ship to the public site.

How to Use It

Three ways in, one behaviour

From a terminal — the shell wrapper auto-starts Ollama if it's asleep:

# search — top-k chunks, each: score, page, heading, snippet
ksearch "how do dealers hedge short gamma"
ksearch "signs of absorption reversal" -k 8

# rebuild the index after editing notes
ksearch-build

Without the shell wrapper, the script runs directly:

python3 ~/.local/share/knowledge-semsearch/semsearch.py "concept" -k 5
python3 ~/.local/share/knowledge-semsearch/semsearch.py build

And inside an AI coding session, a /note-search skill points the assistant at the same script, so it retrieves from my notes by meaning instead of grepping a guessed keyword.

Commands & parameters

ksearch "q"
Search. Positional query string. Prints ranked hits.
-k N
How many hits to return. Default 5.
ksearch-build
Full rebuild of the index from the notes. Overwrites — not incremental. Run after edits.
--src DIR
(build only) Which folder to index. Defaults to the knowledge notes folder.

Reading a result — the leading number is cosine score (higher = closer). Anything above ~0.6 is a solid match; below ~0.45 the notes probably don't cover it — trust that signal instead of forcing a weak hit.

No auto-reindex — the index is a snapshot. Edit or add notes and the old vectors stay until I run ksearch-build again. Stale results almost always mean "forgot to rebuild."

The Resource Gate

Local models are gated, not banned

A heavy local model once ate enough memory to force a hard power-down. So the rule for any AI assistant working on this machine is: before starting a new local model, show its memory footprint against currently-free RAM and get an explicit OK.

It's a gate, not a wall — local inference is fine as long as the resources are there. The embedding model ksearch uses is already on the sanctioned list because it's small and read-only; anything bigger still has to ask first.

What Local Embeddings Unlock

ksearch is one use of a general tool

Once text can become a vector on my own hardware, the same cheap primitive powers a whole family of tools. ksearch is just the first one I needed. The rest are variations on "compare meanings."

◆
Semantic searchWhat ksearch does — find the notes/docs/emails closest in meaning to a question. The backbone of every item below.
◆
RAG (retrieval-augmented generation)Feed the top chunks into a chat model as context so it answers from my notes instead of guessing. Semantic search is the "retrieval" half; ksearch already produces exactly that input.
◆
DeduplicationNear-identical notes sit almost on top of each other in vector space. Flag pairs above a similarity threshold to find things I wrote twice.
◆
Clustering & auto-taggingGroup all chunks by proximity and topics fall out on their own — a map of what my notes are actually about, without me tagging anything.
◆
ClassificationEmbed a few labelled examples, then sort new text by which label it's nearest to. "Is this note about risk, or about a setup?" with no training.
◆
Recommendation / "more like this"Given one note, surface its nearest neighbours — related reading I'd forgotten I had.
◆
Code searchSame trick over a codebase: "where do we handle reconnect logic" finds it by concept, across files that never use that phrase.

The through-line — every one of these is "turn things into vectors, then measure closeness." Build the embedding step once (that's ksearch's build) and each new tool is a different way of reading the same index.

Extending It

Where I'd take it next
→
Point it at more corpora--src already lets build index any folder. Aim it at trade journals, saved articles, or code — one tool, many indexes.
→
Hybrid searchRun grep and semantic together, blend the scores. Exact ticker symbols where grep wins, fuzzy concepts where embeddings win.
→
RerankingTake the top ~20 semantic hits and have a small model re-order them for the specific question — cheap precision boost on top of recall.
→
Full local RAGWire the top chunks into a local chat model and ask questions in plain language — a private assistant that answers only from my own writing.
→
Bigger embedding modelSwap nomic-embed-text for a stronger one if recall ever disappoints — same code, one constant changes, resource gate permitting.
